On the F1 score

The low F1 (0.149) is expected given the 5.0% event rate — strong class imbalance. The model uses class_weight="balanced_subsample" and is evaluated on threshold-tuned predictions. AUC (0.836) is the primary metric: it measures discriminative ability independent of threshold and class distribution. The model reliably separates high-risk days from low-risk ones — which is what matters for the visual alarm system embedded in each painting.

Cross-validation results

Time-series aware 5-fold cross-validation (TimeSeriesSplit). Folds are strictly sequential — no data from the future leaks into any training window. The event rate is 5.0%, addressed with class_weight='balanced_subsample' across 400 estimators.

Feature importance — top 10 (SHAP)

Mean absolute SHAP values computed over 200 training samples via TreeExplainer. Minimum daily pressure ranks first — sharp pressure drops are the clearest precursor of incoming frontal systems. Temperature maximum ranks second, directly defining heat and cold event thresholds. Cosine day-of-year ranks third, confirming that Seville's extreme event distribution is strongly seasonal.

The dry index (thermal range × (100 − mean humidity)) appears as a composite proxy for the conditions preceding Seville's most dangerous summer heat events: wide diurnal temperature swings combined with very low humidity.

Model configuration

RandomForestClassifier(
    n_estimators=400,
    max_depth=10,
    min_samples_leaf=4,
    class_weight="balanced_subsample",   # resamples weights per tree
    random_state=42,
    n_jobs=-1
)

balanced_subsample recomputes class weights for each bootstrap sample independently — better-calibrated than the global balanced option on highly imbalanced datasets. No feature scaling is applied — Random Forest is scale-invariant. The feature list is saved alongside the model (features.pkl) to guarantee column order consistency between training and inference.

ERA5 data pipeline

The dataset covers two distinct historical periods: 1940–1990 (50 years of pre-industrial and early-industrial baseline climate) and 2010–2024 (recent climate with confirmed warming trend). This split allows the model to learn both long-run climatological patterns and the distributional shifts of contemporary extreme events.

01

Reanalysis download

Copernicus CDS API via cdsapi. 7 variables at 06:00 / 12:00 / 18:00 UTC across both periods. Area: 38°N–36.5°N, 6.5°W–5°W (bounding box around Seville). Two NetCDF4 series per year — instantaneous (temperature, pressure, wind components, dewpoint) and accumulated (precipitation, cloud cover).

02

Physical conversion & daily aggregation

K → °C · Pa → hPa · precipitation diff × 1000 (m → mm) · wind components → speed via √(u² + v²) · humidity from dewpoint via Magnus formula. Spatial mean over the bounding box. Resampled to daily: Tmax, Tmin, Tmean, Pmean, Pmin, windmax, windmean, precipsum, humiditymax, humiditymean, cloudmean.

03

Bias correction

ERA5 systematically underestimates daily Tmax in Seville by ~2.5°C, a known artifact of the 1.5° grid resolution in a city surrounded by low terrain. A constant additive correction (+2.5°C) is applied to all temperature fields before feature engineering and before label thresholding.

04

Feature engineering — 39 features

Raw daily aggregates (11) · 3-day and 7-day rolling means for temperature, pressure, wind, humidity (8) · 1/2/3-day lags for temperature, precipitation, wind (9) · first-order gradients for temperature and pressure (2) · day-of-year sine/cosine seasonality (2) · derived composites: thermal range, heat intensity, pressure deficit, humidity range, wind spike, dry index, normalized pressure (7).

05

Label definition

Binary target: extreme event tomorrow (shift −1 day). Four triggers — heat (Tmax ≥ 38°C, AEMET heat wave criterion) · cold (Tmax ≤ 10°C) · wind (speed ≥ 8 m/s) · rain (precip ≥ 1 mm/day). OR logic across all four. Event rate: 5.0% of days.

Production inference

The model was trained on features that require temporal context — rolling means, lags, gradients. A single API snapshot cannot provide these. The system accumulates one daily record in a CSV that persists in the repository, building real feature vectors progressively. When no local history exists, it falls back to the last available ERA5 observation.

Day 0
No history. Fallback to ERA5 CSV — the last observed day from the training dataset is used as the feature vector.
Days 1–6
Partial history. Rolling means use min_periods=1 — functional from day 1, improving progressively. Lag features partially filled with current value as proxy.
Day 7+
Rolling means fully real (ma3 and ma7). Pressure and temperature gradients computed from actual observed history.
Day 14+
All lag features complete. Feature vector is equivalent to training conditions. Full prediction quality.

Three daily API calls (08h, 12h, 20h) aggregate into a single daily record. Deduplication logic ensures exactly one row per calendar day regardless of how many times the workflow runs — matching the ERA5 daily frequency used in training.
